Java Interview Series | Internal Working of HashMap
One of the most frequently asked interview questions is the internal working of HashMap.
Let’s understand it simply--

📌 Map & HashMap Basics
Map is part of the Collection Framework, but it does NOT extend the Collection interface directly.
HashMap is a class that implements the Map interface.
Data is stored in key–value pairs.
Keys must be unique, values can be duplicated.
Insertion order is NOT maintained.
Allows one null key and multiple null values.
 
⚙️ Internal Working of HashMap
A HashMap works using 4 main components:
Key – A unique identifier
Value – Data associated with the key
HashCode – A finite integer value generated from the key
Bucket – An array index where data is stored
➡️ Default bucket array size = 16
 
🧠 How Data Is Stored (Step by Step)
✅ Step 1: Hashing the Key
Every key in Java inherits the hashCode() method from the Object class
This method is overridden by most classes like String, Integer, etc.
When we call put(key, value) internally:
key.hashCode() is invoked
It returns an integer hash value
📌 hashCode() is present in java.lang.Object
 
✅ Step 2: Calculating the Index
HashMap converts the hash code into an array index
Internally, Java uses:
 
index = hash & (n - 1)
(where n is the array size)
 
✅ Step 3: Storing the Entry
Data is stored at the calculated bucket index
Each entry is stored as a Node containing:
hash
key
value
next (address of next node)
 
💥 Collision
Collision occurs when multiple keys are stored in the same bucket.
This is normal behavior in HashMap and is handled internally.
 
💥 Collision Handling
By default, HashMap stores collided entries in a Linked List
From Java 8 onwards, when too many entries are stored in the same bucket:
The Linked List is converted into a Red-Black Tree
This reduces search time from O(n) to O(log n)
 
🔁 When does Linked List convert to Tree?
If more than 8 entries are stored in the same bucket
AND the overall HashMap capacity is at least 64
Then the Linked List is converted into a Red-Black Tree
 
⏱️ Time Complexity
Ideal case: O(1)
Linked List: O(n)
Red-Black Tree: O(log n)

Java Microservices — Real Answers Senior Interviews Expect

1️⃣ Why does increasing traffic reduce throughput?
→ Because contention, context switching, and downstream bottlenecks amplify under load.

2️⃣ Improper thread pool sizing impact?
→ Too small = starvation, too large = CPU thrashing & GC pressure.

3️⃣ Indexes correct but DB slow?
→ Lock contention, connection pool saturation, or I/O bottlenecks.

4️⃣ Horizontal scaling fails when?
→ When state/session/data locality isn’t externalized.

5️⃣ Cascading timeouts happen why?
→ Because synchronous dependencies amplify latency across the chain.

6️⃣ GC pauses even with normal heap?
→ High object churn → frequent minor collections.

7️⃣ Connection pool misconfiguration effect?
→ Threads block waiting → latency spikes without high CPU.

8️⃣ Health checks causing damage?
→ Aggressive probes under load trigger unnecessary restarts.

9️⃣ Message ordering breaks impact?
→ Business inconsistency in event-driven workflows.

🔟 Retry storms happen when?
→ Retries without backoff + shared downstream dependency failure.

1️⃣1️⃣ Autoscaling doesn’t fix performance because?
→ It scales symptoms, not the real bottleneck (DB/cache/network).

1️⃣2️⃣ Detect memory leaks how?
→ Heap dumps + GC logs + object retention analysis.

1️⃣3️⃣ Distributed locks risk?
→ Become a hidden single point of failure.

1️⃣4️⃣ Kafka rebalancing impact?
→ Consumer pause → throughput drop during partition reassignment.

1️⃣5️⃣ Idempotency ensure kaise?
→ Idempotency keys + deduplication storage.

1️⃣6️⃣ Blocking calls in reactive systems?
→ Block event loop → collapse concurrency model.

1️⃣7️⃣ Trace across 15 services?
→ Distributed tracing (correlation IDs + centralized logging).

1️⃣8️⃣ Design APIs for high load?
→ Backpressure, rate limiting, caching, graceful degradation.
